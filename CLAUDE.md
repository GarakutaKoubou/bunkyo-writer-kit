# 文京経済新聞 記事生成セッション

## ミッション
プレスリリースまたは取材文字起こしを受け取り、文京経済新聞の記事を生成してGoogle Docsに保存する。
コメントや写真が不足する場合は、依頼メール文も生成する（パターンA〜C）。

## 参照ファイル（記事生成のたびに必ず読み込むこと）
- `writing_rules.md` ：基本執筆ルール
- `editor_feedback.md`：編集長からの指摘パターン（蓄積）

## 詳細ルール（.claude/rules/ に分割・自動読み込み）
- `safety.md`：情報混入防止・先祖返り防止（3層防御）
- `gdocs.md`：Google Docs保存・更新・Notion同期
- `photo.md`：写真選定・スロット割り当て
- `json-format.md`：JSON出力フォーマット・住所・リンク
- `preview.md`：プレビュー表示・修正ループ・承認後フロー
- `quality.md`：ファクトチェック・品質チェック・PDF対処
- `index.md`：記事インデックス管理・ステータス

---

## 絶対ルール（3つ・違反禁止）

### 1. 情報混入防止（最重要）
- **今回の素材のみ**を根拠にする。過去記事・別記事の情報は一切使わない
- 確認できない情報は書かない（「プレスリリースに記載なし・確認要」と明記）
- 詳細 → `.claude/rules/safety.md`

### 2. コメント語順
- **必ず「主語＋は」を先に、コメントを後に**
  - ✅ 白井会長は「○○したい」と話す。
  - ❌ 「○○したい」と白井会長は話す。

### 3. PDFが読めない場合
- 本文テキストが取得できなかったら**直ちに作業を止め**てユーザーに報告する
- 記憶や推測だけで記事を生成してはならない

### 4. 住所は必ずWebSearchで確認する（推測・類推禁止）
- プレスリリースに番・号まで記載がない場合は、**必ずWebSearchで正式住所を検索**してから書く
- 施設名・地名イメージからの推測は絶対にしない（「椿山荘＝目白台」などの先入観で誤った住所を書いたインシデント発生）
- **住所の粒度ルール：**
  - 東京都内（記事本文リード文）：区名＋丁目まで「（文京区関口2）」
  - 東京都以外（記事本文）：都道府県名＋市区町村まで「（千葉県茂原市）」（町名・丁目は書かない）
  - `address`フィールド（管理システム用）は全国どこでも番・号まで「文京区関口2-10-8」

### 5. 素材・イベントの日付チェック（記事化前に必須）
素材を受け取ったら、記事を生成する前に以下を確認する：

**① プレスリリースの発行日が古い場合**
- 発行日が現在から **6か月以上前** の場合は、記事化を一時停止し、ユーザーに確認する：
  > 「このプレスリリースは〇〇年〇〇月〇〇日付けです。記事化してよいですか？」

**② イベント・展示の開催日が過ぎている場合**
- 素材にイベント開催日・展示期間・申込締切などが記載されている場合、**その日付が現在日付を過ぎていないか**を確認する
- 過ぎていた場合は、記事化を一時停止し、ユーザーに確認する：
  > 「このイベントの開催日（〇〇年〇〇月〇〇日）は既に過ぎています。記事化してよいですか？」
- 複数の日程がある場合は**最終日**を基準にする

**確認なしに記事を生成してはならない。**

## セッション開始時の必須手順

セッション開始時（最初の作業の前に）必ず以下を実行する：

1. `CONTEXT.md` を Read tool で読み込む
2. `## 進行中の記事一覧` を確認し、作業対象の記事と次のアクションを把握する
3. ユーザーに「現在N本が進行中です。〇〇の続きでしょうか？」と状況を伝える

> **CONTEXT.md が存在しない場合**：`python3 update_context.py` を実行して生成する

---

## CONTEXT.md 更新ルール

| タイミング | AIが行うこと |
|-----------|------------|
| 記事作業を**開始**するとき | 該当記事の「次のアクション」列を現在の作業内容に合わせて Edit tool で更新 |
| `index_generator.py` 実行後 | update_context.py が自動実行されるため、「次のアクション」を確認・必要なら修正 |
| ステータスが変わったとき | `python3 update_context.py` を実行（または index_generator.py 経由で自動更新） |
| **セッション終了前** | `## 最終セッションメモ` に今日の日付と作業内容を1〜3行で追記（最新を一番上に） |
| 停滞・保留が発生したとき | `## 停滞・保留メモ` に理由を記入。解決したら削除する |

## セッションの基本フロー

> **WORK_DIR** = `/tmp/bunkyo_YYYYMMDD`（記事ごとの作業フォルダ）

```
① 素材受け取り → mkdir -p /tmp/bunkyo_YYYYMMDD
   ✏️ **執筆中登録（必須・最初に行う）**：
      python3 claim_article.py "記事タイトル（仮）" --work-dir /tmp/bunkyo_YYYYMMDD
   → IDはSheets行番号から自動採番（複数セッション同時実行でも衝突しない）
   → claim_id.txt が作業フォルダに保存される（article.json の id として自動引継ぎ）
   → 他のライターのINDEXに「✏️ 執筆中」として表示され、二重執筆を防ぐ
   ※ **日付チェック（必須）**：プレスリリース発行日が6か月以上前 or イベント開催日が過去 → ユーザーに確認（絶対ルール4参照）
   ※ GoogleフォトアルバムURLがあれば写真を取得
② writing_rules.md + editor_feedback.md を参照して記事を生成
③ コメント有無・写真有無を判定
④ プレビュー表示（詳細 → .claude/rules/preview.md）
   a. article.json に書き出す（idフィールドにclaim_id.txtの値を必ず設定）
   b. 素材フィンガープリント登録（初回のみ）
   c. article_check.py でチェック（必須・毎回）
   d. ファクトチェック結果をチャットに表示（詳細 → .claude/rules/quality.md）
   e. preview_generator.py --open
   f. --save-article + index_generator.py（articles/ に HTML・JSON を自動保存・Sheetsも自動更新）
   → 修正指示 → 修正して④に戻る
   → 「OK」→ ⑤へ
⑤ mail.json 書き出し（コメント/写真不足の場合のみ）
⑥ save_to_gdocs.py で Google Docs に保存（gdocs_url は自動で article.json に書き込まれる）
⑦ --save-article → index_generator.py（Sheetsが自動更新される）
⑧ Google Docs URLをユーザーに返す
```

> **Google Docs 保存は、ユーザーが「OK」と承認してから行う。プレビュー段階では保存しない。**

## ステータス変更の禁止ルール（絶対遵守）

| 操作 | Claudeが行う | 必ず手動で行う |
|------|------------|--------------|
| 下書き保存（`save_to_gdocs.py`） | ✅ status → `draft` | |
| 本番フォルダへ移動（`--move-completed`） | ✅ status → `completed`（📨 申請中） | |
| **公開済みへの変更** | **❌ 絶対にしない** | **✅ ユーザーが手動で `done` に変更** |

- `--move-completed` 実行後のステータスは必ず `completed`（申請中）にする
- `done`（🌐 公開済み）への変更はユーザーが手動で行う。Claudeはいかなる場合も `done` にしない

## article_index.json の扱い（重要）

| ファイル | 役割 | 操作 |
|---------|------|------|
| `Google Sheets` | **正規データ（Single Source of Truth）** | 読み書き両方 |
| `article_index.json` | **出力専用キャッシュ** | 書き込みのみ（index_generator.pyが自動生成） |

- `article_index.json` を直接読み込んでインデックス操作の起点にしてはならない
- インデックスを参照・修正するときは必ずSheetsをMCPツールで確認する
- `index_generator.py` を実行すると Sheets → article_index.json → article_index.html の順で自動生成される

## 記事データの一貫性（3層防御）

| 層 | 仕組み | 防ぐもの |
|----|--------|---------|
| 第1層 | 記事ごとの作業フォルダ `/tmp/bunkyo_YYYYMMDD/` | データ混入 |
| 第2層 | `article_check.py` 自動検証 | 先祖返り・ルール違反 |
| 第3層 | CLAUDE.md + rules ルール | 最後の安全網 |

**共用ファイル（`/tmp/article_preview.json`）は使用禁止。**
詳細 → `.claude/rules/safety.md`

## スキル（.claude/skills/・タスクに応じて自動起動）
- `generate-article`：記事生成フロー（①〜⑧）
- `update-comment`：コメント組み込みフロー
- `upload-gdocs`：Google Docs保存・完成フォルダ移動
- `thank-you-email`：記事公開後のお礼メール生成

## エージェント（.claude/agents/・独立コンテキストで実行）
- `fact-checker`：ファクトチェック専門（article_check.py + 素材照合）

## スキル・エージェントを追加・更新したとき（自動化済み）
- `SKILL.md` または `.claude/agents/*.md` を新規作成・更新すると、**PostToolUseフックが自動で `sync_notion_skills.py` を実行**してNotionの環境構成ドキュメントを更新する
- 手動で同期したい場合：`python3 sync_notion_skills.py`

## 記事の深さ・情報量チェックリスト

記事を書く前に以下を確認する：
- [ ] 展示・イベントの具体的な展示物・プログラム名を全て書いたか
- [ ] 各展示物の「背景・意義・面白さ」を1文以上添えたか
- [ ] プレスリリースの「主な展示内容」「エリア詳細」を全て読んだか
- [ ] 読者が「行きたい・体験したい」と思える描写があるか
- [ ] コメントは原文の「ですます」調を「である」調に変換したか

